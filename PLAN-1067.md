## 1. Feature A — Config Versioning with Rollback

**Goal.** Track every change to `dispatch_config.json`, `forge_config.json`, and per-role prompts (`prompts/*.md`) over time as immutable snapshots in TheForge. Operator can list versions, diff any two, and roll back to any prior version.

### Cross-cutting touchpoints

- `db_migrate.py:733` — register `migrate_v9_to_v10`.
- `schema.sql` after line 688 — append new tables + bump `user_version = 10`.
- `equipa/config.py` — currently centralizes feature flags; add a hook that invokes the snapshotter on first read.
- `equipa/cli.py` and/or `forge_orchestrator.py` — CLI subcommands.
- `prompts/*.md` — read-only inputs to snapshotter.
- `equipa/dispatch.py` — auto-snapshot trigger on dispatch start.

### A1. Schema migration v9 → v10 + storage layer

- **Complexity:** simple
- **Title:** `feat(config-versioning): schema v10 — config_versions + config_version_files tables`
- **Description:** Add migration `migrate_v9_to_v10(conn)` to `db_migrate.py:733` and matching CREATE TABLE blocks at the bottom of `schema.sql:688` (immediately before the `PRAGMA user_version` stamp; bump stamp to 10). Two tables:
  - `config_versions(id INTEGER PRIMARY KEY, project_id INTEGER, created_at TEXT, source TEXT CHECK (source IN ('manual','auto-dispatch','auto-cli')), commit_message TEXT, content_sha TEXT NOT NULL, parent_version_id INTEGER, FOREIGN KEY(project_id) REFERENCES projects(id))` with index `idx_config_versions_project_created`.
  - `config_version_files(id INTEGER PRIMARY KEY, version_id INTEGER NOT NULL, file_path TEXT NOT NULL, content_blob TEXT NOT NULL, file_sha TEXT NOT NULL, byte_size INTEGER, FOREIGN KEY(version_id) REFERENCES config_versions(id) ON DELETE CASCADE, UNIQUE(version_id, file_path))` with index `idx_cvf_version`.
  - Use `content_sha` = SHA-256 over sorted (file_path, file_sha) pairs for dedup; insert no new row if `content_sha` matches the latest existing row for that project.
  - Migration must be idempotent (`CREATE TABLE IF NOT EXISTS`), set `PRAGMA user_version = 10`, and follow the function-signature pattern of `migrate_v8_to_v9` at `db_migrate.py:660`.
  - Tests: `tests/test_db_migrate_v10.py` — applies v9→v10 to a fresh DB, asserts both tables exist, FK CASCADE works, dedup short-circuits on identical content.
- **Dependencies:** none
- **Open Q:** Should `content_blob` be plain text or zstd-compressed? Prompts are tiny (~5 KB total); recommend plain text for v1, defer compression.

### A2. Snapshot / diff / rollback service module

- **Complexity:** medium
- **Title:** `feat(config-versioning): equipa/config_versions.py snapshot+diff+rollback API`
- **Description:** New module `equipa/config_versions.py` with:
  - `snapshot(project_id: int, source: str, commit_message: str | None = None, files: Iterable[Path] | None = None) -> int` — defaults `files` to `[dispatch_config.json, forge_config.json, *prompts/*.md]` (resolved relative to repo root via `equipa.constants`). Computes per-file SHA, computes aggregate `content_sha`, writes one row to `config_versions` + N rows to `config_version_files` in a single transaction via `equipa.db.db_conn`. Returns new `version_id`. Skips write + returns existing id if aggregate hash matches latest.
  - `list_versions(project_id: int, limit: int = 50) -> list[dict]`.
  - `diff(version_a_id: int, version_b_id: int) -> dict[str, str]` — returns `{file_path: unified_diff_string}` using stdlib `difflib.unified_diff`.
  - `rollback(version_id: int, dry_run: bool = False) -> list[Path]` — rewrites each tracked file to its stored `content_blob`. If `dry_run`, returns file list without writing. Always takes a fresh `snapshot(source='auto-rollback')` of the pre-rollback state first so rollback itself is reversible.
  - All file writes use atomic-rename pattern (`tmp + os.replace`).
  - Path-traversal guard: refuse any `file_path` outside the repo root (resolve + `Path.is_relative_to`).
  - Tests `tests/test_config_versions.py`: snapshot dedup; diff round-trip; rollback round-trip; rollback writes pre-snapshot first; refuses traversal.
- **Dependencies:** A1
- **Open Q:** Should `rollback` refuse if any tracked file has uncommitted edits relative to the most recent snapshot? Default recommendation: warn + require `force=True`.

### A3. CLI surface + auto-snapshot on dispatch start

- **Complexity:** medium
- **Title:** `feat(config-versioning): CLI commands + auto-snapshot hook on dispatch`
- **Description:** Add CLI subcommands to `equipa/cli.py` (or `forge_orchestrator.py` if that owns the CLI surface — consult `equipa/cli.py:main` first):
  - `equipa config snapshot [--project N] [-m MSG]`
  - `equipa config list [--project N]`
  - `equipa config diff <id_a> <id_b>` (prints unified diff)

  Add a single hook in the dispatch entry path (look for the function that loads `dispatch_config.json` near the top of `equipa/dispatch.py` or the CLI handler) that calls `config_versions.snapshot(project_id, source='auto-dispatch')` exactly once per dispatch invocation. The hook must be cheap (dedup on hash means no writes if config unchanged) and must not crash dispatch on snapshot failure (wrap in try/except + log).
  - Tests: `tests/test_cli_config.py` exercises each subcommand against a temp DB; integration test that two consecutive dispatches with unchanged config produce only one new row.
- **Dependencies:** A1, A2
- **Open Q:** Should auto-snapshot also fire on heartbeat sweep so out-of-band edits get captured? Recommend yes — add 1-line call to `equipa/heartbeat.py:_run_sweep` once A3 lands.

## 2. Feature B — Session Persistence Across Heartbeats / Cycle Boundaries

**Goal.** Today, an agent run that hits an orchestrator cycle boundary (heartbeat re-dispatch, flow revision bump, manager loop tick) starts with a clean prompt unless soft-checkpoint compaction recovery happens to fire. Extend persistence so that *every* cycle boundary preserves: open files, partial reasoning, recent tool calls — mirroring the within-task soft-checkpoint contract but at the orchestrator level.

### Cross-cutting touchpoints

- `equipa/checkpoints.py:107` (`save_soft_checkpoint`) — extend, do not replace; cycle persistence is a superset.
- `equipa/heartbeat.py` — sweep loop is the natural boundary detector.
- `equipa/flows.py:VALID_FLOW_STATES` — flow revision bumps are also boundaries.
- `equipa/loops.py` — primary dispatch loop; needs hooks at cycle entry/exit.
- `equipa/agent_runner.py` — where prompt assembly happens; needs to consume restored session state.
- Schema: new `agent_sessions` table (extends but does not replace `agent_runs`).

### B1. Schema + persistence layer

- **Complexity:** simple
- **Title:** `feat(session-persistence): schema v10 — agent_sessions table`
- **Description:** Migration `migrate_v9_to_v10` (or v10→v11 if Feature A lands first; coordinate via dispatch order). Table:
  - `agent_sessions(id INTEGER PRIMARY KEY, task_id INTEGER NOT NULL, role TEXT NOT NULL, project_id INTEGER NOT NULL, cycle_id TEXT NOT NULL, state_json TEXT NOT NULL, byte_size INTEGER, created_at TEXT, last_seen_at TEXT, expires_at TEXT, FOREIGN KEY(task_id) REFERENCES tasks(id), FOREIGN KEY(project_id) REFERENCES projects(id))` with indexes on `(task_id, role, last_seen_at DESC)` and `expires_at`.
  - `state_json` shape (stable contract): `{ "open_files": [...], "files_changed": [...], "files_read": [...], "recent_tool_calls": [{tool,args_hash,ok,turn}], "partial_reasoning": "...truncated to 8 KB...", "turn_count": N, "compaction_count": N, "soft_checkpoint_path": "..." }`. Reuse the soft-checkpoint dict as a subset; this is a strict superset.
  - `cycle_id` is heartbeat-tick UUID or flow-revision string — opaque to persistence layer.
  - Tests: migration applies cleanly; FK CASCADE on task delete is opt-out (we want sessions to survive task purge for postmortem — confirm with operator).
- **Dependencies:** none (or A1 if Feature A ships first — coordinator picks)
- **Open Q:** Default expiry — 7 days? 30? Recommend 14 days w/ heartbeat purge.

### B2. Capture / restore service + soft-checkpoint bridge

- **Complexity:** complex
- **Title:** `feat(session-persistence): equipa/sessions.py capture+restore at cycle boundaries`
- **Description:** New module `equipa/sessions.py`:
  - `capture(task_id, role, project_id, cycle_id, *, soft_checkpoint_path: Path | None) -> int` — assembles `state_json` from: most recent soft checkpoint (`load_soft_checkpoint` at `equipa/checkpoints.py:168`), `.forge-state.json` if present, and the live tool-call ring buffer (will need a small additive change to `equipa/agent_runner.py` to expose the buffer — keep the surface tiny: one accessor).
  - `restore(task_id, role) -> dict | None` — returns most recent non-expired `state_json` for the task+role.
  - `build_resume_prompt(state: dict) -> str` — extends `build_compaction_recovery_context` at `equipa/checkpoints.py:192` (do not duplicate; refactor to share). Output is the prefix prepended to the next agent invocation.
  - `purge_expired() -> int` — returns rows deleted; called by heartbeat.
  - Tests: `tests/test_sessions.py` — round-trip capture/restore; restore returns most recent; expired sessions skipped; resume prompt contains all soft-checkpoint fields.
- **Dependencies:** B1
- **Open Q:** When the agent has BOTH a within-task soft checkpoint AND a cross-cycle session record, which wins? Recommend session record (it's the superset) but include a fallback path so the existing compaction-recovery test suite doesn't regress.

### B3. Wire-in to loops + heartbeat + flow-revision boundary

- **Complexity:** complex
- **Title:** `feat(session-persistence): wire capture/restore into loops, heartbeat, and flow revisions`
- **Description:** Three integration sites, each guarded by a feature flag in `dispatch_config.features` (new key `session_persistence`, default false):
  1. `equipa/loops.py` — at the dispatch loop's per-task try/finally, call `sessions.capture(...)` on exit and `sessions.restore(...)` on entry (passing restored state into the prompt builder via `build_resume_prompt`).
  2. `equipa/heartbeat.py` — add a fourth sweep stage `purge_expired_sessions` mirroring the existing `_run_sweep` pattern; respect `dry_run`.
  3. `equipa/flows.py` — when `transition()` bumps a revision, call `sessions.capture(cycle_id=f"flow:{flow_id}:{revision}")` for every running child task in that flow.
  - Tests: integration test that simulates two heartbeat cycles for a single long task and asserts the second cycle's prompt prefix contains the first cycle's `files_changed` set.
  - Add operator docs to `docs/` describing the flag and the resume-prompt shape.
- **Dependencies:** B1, B2; soft coordination with Feature A (no hard dep)
- **Open Q:** Flow-revision capture could be very chatty (every state transition). Recommend rate-limiting to once per (flow_id, task_id) per 60s.

## 3. Feature C — Portable Project Templates

**Goal.** Export a project as a single archive (`.equipa-template/` directory or `.tar.gz`) and import it into another EQUIPA install — including `projects`, `tasks`, `decisions`, `session_notes`, `open_questions`, `lessons_learned`, plus `CLAUDE.md` and any local-only assets. **Adapter-agnostic** schema so a non-Claude agent runtime can ingest the same archive.

### Cross-cutting touchpoints

- `equipa/db.py` — read-only access to all tables.
- `schema.sql` — source of truth for column lists per table.
- `equipa/cli.py` / `forge_orchestrator.py` — CLI surface.
- `docs/` — adapter spec lives here.
- Project working directory — source of `CLAUDE.md` and per-project assets (mapping comes from `forge_config.project_dirs`).

### C1. Adapter-agnostic manifest spec + read-side exporter

- **Complexity:** medium
- **Title:** `feat(templates): equipa/templates.py exporter + manifest schema`
- **Description:** New module `equipa/templates.py`:
  - `export(project_id: int, dest_dir: Path) -> Path` — writes the following layout:
    ```
    dest_dir/
      manifest.json          # version, exported_at, source_runtime='equipa-py', table list, row counts, sha256 of each file
      tables/
        projects.jsonl
        tasks.jsonl
        decisions.jsonl
        session_notes.jsonl
        open_questions.jsonl
        lessons_learned.jsonl
      assets/
        CLAUDE.md            # if present in project working dir
        prompts/             # per-project prompt overrides if any
        ...
    ```
  - JSONL rows use **stable, runtime-neutral field names** (snake_case, no Python-specific types). Foreign keys stay as integer IDs but the manifest declares `id_namespace='source'` so importer can remap.
  - Manifest schema documented in new file `docs/template-spec.md` — this is the contract non-Claude adapters must honor. Spec must list: required tables, optional tables, required manifest fields, hashing algorithm, FK remapping rules.
  - Tar option: `export(..., archive=True)` produces `dest_dir.tar.gz`.
  - Tests: round-trip the manifest schema validator on sample data; assert no Claude-specific fields (no `claude_session_id`, no `model_*` keys with Claude model names) leak into the manifest.
- **Dependencies:** none (independent of Features A and B)
- **Open Q:** Should we redact API keys in `agent_runs`/`api_keys` tables? Recommend **never include `api_keys` table** and scrub `cost_usd` from `agent_runs` if operator passes `--scrub-costs`.

### C2. Importer with FK remapping + collision policy

- **Complexity:** complex
- **Title:** `feat(templates): equipa/templates.py importer + FK remapping`
- **Description:** Same module:
  - `import_archive(source: Path, target_project_name: str | None = None, *, on_conflict: str = 'rename') -> int` — returns new `project_id`.
  - Order of operations: validate manifest hash → open transaction → insert new `projects` row → walk each `tables/*.jsonl` in dependency order → maintain an `id_remap: dict[(table_name, source_id), new_id]` so foreign keys (e.g., `tasks.project_id`, `decisions.resolved_by_task_id`) get rewritten before insert.
  - Conflict policy `on_conflict='rename'` appends `-imported-N` to project name; `'merge'` requires explicit target project_id; `'fail'` aborts.
  - Asset files copied into the target project's working directory (path resolved via `forge_config.project_dirs[new_project_name]`); refuse to overwrite if destination has files (require `--force`).
  - Tests: full round-trip — export project_id=23 to tmp dir → import into a fresh DB → assert task count, decision count, lesson count match; assert FKs are valid in target DB.
- **Dependencies:** C1
- **Open Q:** Should `lessons_learned` rows with embeddings be re-embedded on import (different ollama model on target host) or imported as-is? Recommend import as-is + flag for re-embed.

### C3. CLI + adapter compliance test fixture

- **Complexity:** medium
- **Title:** `feat(templates): CLI commands + non-Claude adapter compliance tests`
- **Description:** CLI surface in `equipa/cli.py`:
  - `equipa template export <project_id> [--out PATH] [--archive] [--scrub-costs]`
  - `equipa template import <archive_path> [--name NAME] [--on-conflict rename|merge|fail]`
  - `equipa template validate <archive_path>` — runs the manifest validator only, useful for CI.
  Plus `tests/fixtures/template-non-claude/` — a minimal archive that mimics what a hypothetical non-Claude runtime would emit (no `prompts/` dir, different `source_runtime`). The importer must accept it without complaint. This is the **adapter-agnosticism regression test**.
  - Tests: CLI exit codes; `validate` rejects malformed manifest; `import` accepts the non-claude fixture.
- **Dependencies:** C1, C2
- **Open Q:** Where does the spec live for external authors? Recommend `docs/template-spec.md` mirrored to README + linked from `CHANGELOG.md`.

## 6. Recommended Dispatch Strategy

**Wave 1 (parallel, 3 dispatches):** A1, B1, C1.
- All schema / data-shape work. Independent except for the v10/v11 coordination already covered. Each is "simple" or "medium" complexity, fits a single complex-budget cycle easily.

**Wave 2 (parallel, 3 dispatches):** A2, B2, C2.
- Service-layer work that depends on Wave 1 of its own feature. Fully parallel across features.

**Wave 3 (parallel, 3 dispatches):** A3, B3, C3.
- Integration + CLI surface. Two of the three (A3, B3) touch shared modules (`heartbeat.py`, `cli.py`) — see "Heartbeat callback fan-out" risk above. If a conflict materializes, serialize A3 → B3.

**Recommended priority across features (if forced to pick a single order):**
1. **Feature A first.** Lowest risk, highest immediate operator value, isolated to two configs + prompt files. Validates the v10 migration pattern before B1 ships.
2. **Feature C second.** Independent of A and B. Unblocks the "fork an EQUIPA project" use case and provides the adapter-agnosticism fixture other tooling can build against.
3. **Feature B last.** Highest complexity, highest blast radius (`loops.py`, `agent_runner.py`, `heartbeat.py`, `flows.py` all touched). Lands on the most stable base after A and C are in.
